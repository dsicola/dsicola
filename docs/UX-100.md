# UX — Objetivo 100% (nível institucional)

Checklist para garantir experiência de utilizador profissional em todo o DSICOLA.

Referência: [ROADMAP-100.md](./ROADMAP-100.md) (secção 3) e [PRIORIDADE-100.md](./PRIORIDADE-100.md).

---

## 1. ⏳ Loading em tudo

**Regra:** Em toda a ação que demore (submeter formulário, guardar, eliminar, carregar lista), o utilizador deve ver que algo está a acontecer.

| Estado | Item |
|--------|------|
| ☑ | Botões de submissão mostram estado de loading (ex.: “Processando...”, “Salvando...”, “A guardar...”) |
| ☑ | Botão desativado durante o pedido (evitar duplo clique) |
| ☐ | Listagens e tabelas mostram skeleton ou spinner enquanto carregam |
| ☐ | Navegação entre páginas com indicador de carregamento quando aplicável |

**Implementado:** `Button` em `@/components/ui/button.tsx` aceita `loading` e `loadingLabel` (ex.: `loadingLabel="Salvando..."`). Usado em Gestão Financeira (Gerar Mensalidades) e Configurações Instituição (Salvar). Para outros formulários: `<Button loading={isPending} loadingLabel="Processando..." onClick={...}>Guardar</Button>`.

---

## 2. ✅ Feedback visual

**Regra:** O utilizador deve saber imediatamente se a ação teve sucesso, falhou ou requer atenção.

| Estado | Item |
|--------|------|
| ☐ | **Sucesso** — Toast/notificação verde (ex.: “Guardado com sucesso”, “Aluno criado”) após criar/editar/eliminar |
| ☐ | **Erro** — Toast vermelho com mensagem clara (ex.: “Não foi possível guardar. Tente novamente.”) e, se possível, detalhe do erro |
| ☐ | **Alerta** — Toast amarelo/laranja para avisos (ex.: “Dados incompletos”, “Confirme antes de continuar”) |
| ☐ | Fluxos críticos (login, matrícula, pagamento, lançamento de notas) têm feedback consistente em todos os passos |

**Implementação sugerida:** Sistema centralizado de toasts (ex.: Sonner, React Hot Toast, ou componente interno) usado em todas as mutações (create, update, delete).

---

## 3. 📱 Responsivo

**Regra:** O sistema deve funcionar bem em **computador**, **tablet** e **telemóvel**.

| Estado | Item |
|--------|------|
| ☐ | **Computador** — Layouts principais (dashboard, listagens, formulários) usáveis em 1024px+ |
| ☐ | **Tablet** — Navegação e tabelas adaptadas (ex.: scroll horizontal ou cards em vez de tabela larga); menus colapsáveis |
| ☐ | **Telemóvel** — Navegação móvel (menu hamburger, botões grandes), formulários em coluna única, listas scrolláveis; evitar tabelas com muitas colunas em ecrã pequeno |
| ☐ | Páginas principais validadas: Admin, Secretaria, Professor, Aluno, POS, relatórios |

**Implementação sugerida:** Breakpoints consistentes (ex.: 640, 768, 1024, 1280px); testes manuais ou E2E em viewports pequeno/médio/grande.

---

## 4. 📭 Empty states

**Regra:** Quando não há dados para mostrar, não mostrar apenas uma lista vazia. Mostrar mensagem clara e, quando fizer sentido, uma ação sugerida.

| Estado | Item |
|--------|------|
| ☑ | Listagens (alunos, professores, matrículas, mensalidades, turmas, etc.) têm empty state com ícone/mensagem (ex.: “Ainda não há alunos. Adicione o primeiro.”) |
| ☑ | Botão ou link de ação no empty state quando aplicável (ex.: “Criar aluno”, “Registar pagamento”) |
| ☐ | Mensagens de “nenhum resultado” em buscas/filtros (ex.: “Nenhum resultado para ‘X’. Tente outros filtros.”) |
| ☐ | Páginas de relatórios ou dashboards com zero dados mostram mensagem amigável em vez de gráficos vazios sem explicação |

**Implementado:** Componente `EmptyState` em `@/components/ui/empty-state.tsx` (ícone inbox/search, título, descrição, `actionLabel` + `onAction`). Usado em: Gestão Financeira (mensalidades vazias → “Gerar Mensalidades”), AlunosTab (lista vazia → “Criar aluno”), e `ResponsiveTable` quando se passam `emptyTitle`, `emptyDescription`, `emptyActionLabel`, `onEmptyAction`.

---

## Resumo

| Área | Objetivo |
|------|----------|
| Loading | Botão mostra “Processando...” / “Salvando...”; listas com skeleton/spinner |
| Feedback | Toasts sucesso (verde), erro (vermelho), alerta (amarelo) em fluxos críticos |
| Responsivo | Funcionar bem em computador, tablet e telemóvel |
| Empty states | Mensagem clara + ação sugerida quando não há dados |

Quando todos os itens estiverem cobertos, a UX está alinhada ao objetivo 100% para nível institucional.
